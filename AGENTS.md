# AGENTS.md

本文件用于给 coding agent 提供项目级强约束。
核心目标是把“优质架构、代码洁癖、工程质量、零退化交付”内化为本项目的日常开发准则；不要照搬外部提示词里的角色扮演、审批状态机或冗长模板。

## Communication

- 默认使用简体中文回复。
- 专有名词、接口名、类名、变量名、命令保持英文；标准软件工程术语可保留英文以避免歧义。
- 结论要直接、可执行；未验证的内容必须明确说明。
- 对代码、架构、运行态和文档的判断必须锚定本仓库事实，不凭空假设功能或约束。

## Operating Mode

- 默认工作方式是：理解需求和现状 -> 给出必要计划或 task spec -> 直接实现 -> 运行验证 -> 汇报结果。
- 不采用“每个阶段都等待用户确认”的锁定流程；除 Stop Conditions 外，不把计划、架构草案或 TODO 表当作硬阻断。
- 当需求边界清楚时，即使任务较复杂，也应在同一轮内完成计划、实现和验证。
- 只有在需求存在阻断性歧义、会改变架构/数据/API 契约且用户未确认、需要生产凭据/外部 SaaS 写入/破坏性操作、或当前工作树存在会被覆盖的用户改动时，才暂停询问。
- 用户明确要求“只分析/只出方案/不要改代码”时，严格停止在分析或方案层。

## Engineering Quality North Star

- 交付目标不是“能跑就行”，而是可维护、可扩展、可验证、可回滚、可解释。
- 架构质量体现在边界清楚、依赖方向稳定、契约显式、错误模型统一、状态变化可追踪。
- 代码洁癖体现在命名准确、结构克制、重复可控、无死代码、无含糊注释、无魔法值、无吞异常。
- 工程质量体现在测试和构建能证明行为、日志和指标能定位问题、变更不会无声破坏既有能力。
- 性能和可靠性不是后补项；涉及数据库、队列、缓存、外部依赖、并发状态时，设计阶段就要纳入取舍。

## Work Priority

默认优先级：

1. 理解当前代码、文档、配置和真实运行状态。
2. 做最小必要修改，并保持已有功能不退化。
3. 运行相关测试、构建或 smoke 验证。
4. 在最终回复中说明改动、验证结果和遗留风险。
5. 只有在需要沉淀长期决策时才更新文档。

## Task Levels

按任务影响范围选择流程：

- S0 快速任务：
  - 适用：单文件修复、小 bug、明确命令、简单排查、文案调整。
  - 流程：直接查、改、跑、验。
  - 不创建或更新 `docs/`、`tasks/`。

- S1 小功能：
  - 适用：多文件但低风险，不改架构、不改数据契约、不影响部署。
  - 流程：先做简短计划，再直接实现。
  - 如需记录，只在 `tasks/current.md` 增加一条短记录。

- S2 正常功能：
  - 适用：跨模块功能、API 变化、数据库字段、权限、前后端联动。
  - 流程：实现前先写一个 compact task spec，然后直接按 spec 开工，除非触发 Stop Conditions。
  - task spec 只需包含：目标、范围、非目标、影响文件、验收标准、测试方式。

- S3 长线/高风险任务：
  - 适用：架构迁移、微服务边界、生产部署、数据迁移、安全/合规、长期规划。
  - 流程：需要 `docs/requirements/`、`docs/decisions/` 或 ExecPlan；如风险边界已由用户明确，可计划后继续推进。
  - 这类任务才使用父任务、子任务、状态归档和完整回写。

## Requirement And Analysis Rules

- 用户提供的 SRS、用户故事、设计文档、决策记录和接口文档是优先真相来源。
- 信息不足但不阻断时，可以声明合理假设并继续；信息严重不足、相互矛盾或会影响契约时，先提出少量关键问题。
- 需求分析应覆盖：业务目标、用户角色、核心痛点、成功标准、主要流程、异常流程、权限规则、状态转换、数据校验和验收标准。
- 需求文档服务于交付，不为流程而流程；普通开发只保留能指导实现和验收的最小文档。

## Architecture Quality Rules

- 优先沿用本项目已有技术栈、模块边界、目录组织和 helper；不新增平行体系。
- 架构设计必须说明关键取舍，尤其是开发效率、部署复杂度、可维护性、性能、安全、可靠性和扩展性。
- 前端以现有 React/Vite/RTK、Next.js 或项目实际栈为准；后端以现有 Java/Spring Boot/Gradle 模块边界为准。
- 新增跨模块能力时，先明确 API 契约、数据模型、权限边界、错误模型和回滚/兼容策略。
- 设计决策必须贴合当前仓库，不照搬通用模板，不凭空引入未使用框架。
- 模块边界要高内聚、低耦合：Controller 不承载业务规则，Service 不泄露持久化细节，Repository 不夹带业务决策。
- 领域规则应靠近领域对象或清晰的领域服务；跨聚合流程用应用服务编排，避免把所有逻辑堆进单个大方法。
- 读写模型复杂度明显不同时，可以采用 CQRS 思路拆开 Command 与 Query，但不得为概念纯度制造额外层级。
- API、消息、缓存 key、数据库字段、状态机枚举都属于契约；变更时要考虑兼容、迁移和观测。

## Code Cleanliness Rules

- 命名必须表达业务含义；避免 `data`、`item`、`temp`、`handle` 这类无法承载上下文的泛名，除非范围极小。
- 函数应短小且只做一件事；当分支、循环、异常处理和副作用交织时，优先拆出私有方法或领域对象。
- 不保留注释掉的旧代码、调试输出、无用 imports、无用状态、重复配置和未使用依赖。
- 不用注释解释显而易见的代码；注释只解释复杂业务规则、并发边界、性能取舍、迁移兼容点和非直觉决策。
- 禁止用复制粘贴扩散业务规则；重复出现的校验、映射、状态转换、错误处理应收敛到已有 helper 或新的局部抽象。
- 不把异常吞掉；捕获异常必须转换为明确错误、补充上下文日志、触发补偿或向上抛出。
- 不用魔法字符串/数字承载业务语义；优先枚举、常量、配置或类型约束。
- 遵循 SOLID、DRY、DDD、TDD、设计模式等原则，但只在能降低复杂度或匹配项目模式时使用，避免过度设计。

## Production Quality Rules

- 代码应符合生产级要求：边界校验、异常处理、日志与可观测性、必要测试、可回滚思路。
- 关键逻辑必须考虑空值、脏数据、重复请求、并发写入、网络超时、外部依赖失败和重试幂等。
- 不为“容易写”而降低质量；实现要能通过构建、测试或可说明的 smoke 验证。
- 对新增能力要补齐成功路径、失败路径、边界输入和权限/状态约束；只测 happy path 不算完整。
- 对异步流程、队列任务、缓存状态和重试补偿，要能说明状态如何推进、失败如何恢复、重复消息如何处理。
- 对安全相关代码，默认考虑认证、授权、输入校验、敏感日志脱敏、CSRF/XSS/SQL 注入等风险。

## Anti-Regression Rules

- 修改前先检查当前工作树，不能覆盖用户已有改动。
- 修改既有代码前做“保留清单”心智检查：imports、配置项、UI tabs、方法、业务规则、权限逻辑、错误处理和测试入口默认都要保留。
- 严禁 silent deletion：除非用户明确要求，不删除、不简化、不折叠未提及的功能。
- 修 bug 时锁定无关维度：修 API 不改 UI 视觉；修样式不改业务逻辑；修布局不改颜色和字体，除非任务要求。
- 不做无关重构；重构必须服务于当前目标，并证明不会改变行为。
- 对较大改动，先读调用方和测试再动手；不能只改被报错的一行而忽略上游契约和下游消费者。

## Code Delivery Rules

- 在真实文件中修改时，使用最小 diff；不要为了“全量重写”扩大变更面。
- 在聊天中展示代码时，不能使用 `// ... existing logic`、`// ... same as before`、`/* omitted */` 等省略式占位。
- 若用户要求可直接复制的完整文件，则必须输出完整、可运行、无省略的文件内容，并标明文件路径。
- TypeScript 项目中尽量避免 `any`；优先使用明确类型、`unknown`、泛型、接口或 schema 推导。
- UI 文案默认中文；变量、函数、类名使用英文并遵循项目命名习惯。
- 只要完成代码、配置、部署、数据、脚本或文档变更，最终回复必须编写详细 changelog；不要只写“已修复”或泛泛总结。
- changelog 至少覆盖：变更分类、关键文件/模块、行为变化、兼容性影响、验证结果、部署/运行态状态、遗留风险和建议回滚点。
- changelog 应按用户可读的业务/运维视角组织，同时保留必要的类名、接口名、配置 key、commit id、镜像 tag、Pod/API 验证等工程证据。
- 如果某次任务没有产生实际变更，也要明确写出“无代码/配置变更”，并说明本次只做了哪些排查或验证。

## Backend And Data Rules

- Spring bean 统一使用构造器注入，生产类必须通过单一构造器让 Spring 自动注入；严禁使用 `@Autowired`（包括构造器、字段和 setter）。
- 需要可选依赖、测试替代依赖或默认运行时对象时，优先使用 `ObjectProvider`、明确的 `@Bean` 配置、test configuration 或局部工厂方法；不要通过多个生产构造器配合 `@Autowired` 解决。
- 数据库访问必须做成本取舍：JOIN、Batch Query、DTO Projection、分页、游标/流式处理应根据基数和访问模式选择。
- M:N 或高基数 1:N 关系默认避免大 JOIN，优先 two-phase loading：先查 root/paged IDs，再 `WHERE IN` 批量查关联数据，在应用层组装。
- 报表/看板类复杂读取优先使用扁平 DTO Projection，只选择必要列，避免加载完整托管实体。
- 严格 1:1、低数据量 lookup/static dictionary 场景可以 JOIN，但需要确认基数稳定。
- 事务范围要“必要但尽量短”：事务内不得包含 HTTP/RPC 调用、文件 I/O 或长耗时计算；纯读默认不加事务，除非隔离级别或 LOB 等场景需要。
- 单聚合内可使用强一致事务；跨聚合、跨服务或外部系统更新优先考虑领域事件、消息队列、后台任务和最终一致性。
- 处理大数据集时优先分页、cursor 或 streaming；避免一次性加载不可控数据进入内存。

## Frontend Rules

- 前端改动必须尊重现有视觉和交互风格；除非用户明确要求 redesign，不主动改变整体 UI 气质。
- 新页面或大功能先明确组件结构、状态归属、数据流、错误态、加载态、空态和权限态。
- React 代码中优先使用明确 Props/API response 类型；事件类型使用具体 React event type。
- 全局状态仅用于跨页面/跨模块共享状态；局部交互优先组件本地状态或现有表单/查询方案。
- 避免引入不必要动画和视觉噪声；若做面向用户的前台站点或品牌页，可强化记忆点、动效和响应式体验。
- 所有前端任务交付前尽量运行 `npm run build`、`npm test`、lint 或本项目已有验证命令；必要时做浏览器 smoke 验证。
- 前端质量不只看页面能显示：还要检查表单校验、接口错误、loading 防抖、空数据、超长文本、窄屏布局、权限隐藏和刷新后的状态恢复。

## Public Web Quality Rules

- 仅在面向公开访问、搜索收录或品牌传播的页面上应用公开站点质量规则；后台管理、内网工具和非公开页面不强行套 SEO。
- 公开页面要检查语义化 HTML、heading 层级、metadata、canonical、Open Graph、robots、sitemap、结构化数据和 Core Web Vitals 风险。
- Next.js 公开页面优先使用 App Router 原生能力、Server Components、`generateMetadata`、`next/font`、`next/image` 和 JSON-LD。
- 内容策略必须兼顾 E-E-A-T、真实用户意图、品牌记忆点和技术可抓取性；不做关键词堆砌。

## Documentation Rules

- 不要为了满足流程而重复描述代码已经表达清楚的内容。
- 文档只记录：
  - 新需求边界
  - 架构决策
  - 数据/API 契约
  - 迁移步骤
  - 验证结果
  - 遗留风险
- 普通任务完成后，不强制移动 task 文件。
- 长线任务完成后，才同步 `tasks/backlog/`、`tasks/in-progress/`、`tasks/done/`。

## Repository Map

- `docs/requirements/`：长期需求或高风险需求。
- `docs/decisions/`：架构决策记录。
- `docs/reports/`：调查报告、性能分析、事故分析。
- `docs/migrations/`：迁移记录。
- `tasks/current.md`：轻量当前任务板。
- `tasks/backlog/`、`tasks/in-progress/`、`tasks/done/`：仅用于 S3 长线任务。

## Verification Rules

- 交付前尽量运行最相关的测试、构建、lint、HTTP/API smoke、浏览器 smoke 或运行态检查。
- 无法运行验证时，必须说明原因、未验证范围和建议的后续验证命令。
- 涉及生产候选、GitOps、队列、缓存、异步任务或运行态修复时，不能只看本地构建；应尽量补充线上同步、Pod/日志/API/队列状态等证据。
- 对 review 类任务，优先列风险和缺陷，按严重度排序，并给出文件/行号证据和可执行修复建议。

## Subagents

- 只有在任务边界清晰、写入范围不重叠、结果可独立验证时使用 subagents。
- subagent 适合调查、并行实现独立模块、测试验证。
- subagent 返回内容应简短：结论、关键文件、验证命令、风险。
- 主 agent 负责集成、冲突处理和最终验收。

## Stop Conditions

遇到以下情况先暂停并说明：

- 需求边界会影响架构或数据契约，但用户没有确认。
- 需要生产凭据、外部 SaaS 写入或破坏性操作。
- 当前工作树存在会被本任务覆盖的用户改动。
- 用户明确要求先确认方案、只做审查、只做文档或不要改代码。
