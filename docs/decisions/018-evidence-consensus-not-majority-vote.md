# ADR-018: 证据处理采用候选加独立验证

## 状态

Accepted

## 决策

证据准备分为确定性规范化、异构 AI 清洗审查、异构 AI 压缩候选和独立验证四层。AI 节点不覆盖原始文档；最终验证器必须回看原文并检查候选中的遗漏、歪曲和无来源断言。通过验证的单一结果写入不可变快照，全部中间结果写入审计表。

推理强度按模型能力而不是统一字符串配置。`gpt-5.6-sol` 和实测接受 `MAX` 的 Gemini 使用 `MAX`；普通 GPT 和 Grok 使用其已验证上限 `XHIGH`。供应商不支持的等级不得靠自动降级静默掩盖。

AI 节点采用逻辑角色和席位分层。`roleTemplateId` 是逻辑角色身份，保存共享目标和 Prompt；每个 `WorkflowNodeDefinition` 是一个可独立运行、重试、Fallback 和审计的席位，只绑定一个主 Provider/模型。相同角色可以有多个异构席位，但不得把多个 Provider 藏进一个节点内部。主席保留最终裁决权，席位数量不直接等于投票权或置信度。

Provider 插拔通过 `ai_provider_profile`、`ai_model_profile` 和 `AiRuntimeBindingResolver` 完成。Profile 描述协议、推理参数样式、Base URL/Secret 引用、超时、模型启用状态和最大推理等级；通用 Chat/Responses Gateway 负责线上调用。业务应用层不判断厂商名称。

## 原因

- 多数模型可能共享训练偏差，多数投票不能证明事实正确。
- 直接让 AI 改写规范化文档会破坏证据可追溯性。
- 独立候选能暴露遗漏差异，最终验证器回看原文才能形成可审计共识。
- 工作流节点继续承担模型、提示词、重试和上下文配置，避免在服务代码里固化厂商。
- 一个席位一个 Provider 使失败隔离、成本核算和 A/B 对比保持可解释。

## 后果

- 每个文档的 LLM 调用数和延迟增加，需要依赖已有预算、超时、重试与 Worker 背压。
- `EvidenceSnapshot` 仍只有一个稳定入口，下游双环境分支无需改变。
- 历史 v5 不修改；v6 发布后可以通过归档 v6、重新发布 v5 回滚默认拓扑。
- 新增或下架兼容 Provider 主要是 Profile、模型能力和 Secret 配置变更，不要求修改研究业务服务。
