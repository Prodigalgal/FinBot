# ADR-004：版本化 Council Workflow DAG

## 状态

已接受，2026-07-11。

## 背景

现有 Council 通过 `roles + phases + chair` 描述配置，调度器只支持整组并行或整组轮流。用户需要像工作流一样直接操作节点和连线，并要求每个节点独立配置 LLM 与提示词。如果把完整角色配置复制到可视化节点中，会形成 role 与 node 两套配置源；如果连线只保存布局，则无法改变真实辩论行为。

## 决策

- `roles`、`chair` 继续作为角色及 LLM 配置的唯一真相源。
- 新增 `workflow`，只保存节点类型、role 引用、位置和有向边。
- 后端 `CouncilTemplate.from_payload` 是最终校验边界，前端校验只用于提前反馈。
- 调度器按 DAG 拓扑层运行。跨轮消息广播以保持真正多轮辩论；同轮消息沿工作流依赖传递。
- 不配置 workflow 的旧模板由后端生成兼容图，API 返回时即包含显式 workflow。
- 默认产品委员会使用风险汇聚结构；通用旧模板默认使用 input 到所有角色、所有角色到 chair 的扇出/汇聚结构。

## 取舍

- 继续保留 phase，使用户能够同时表达“角色依赖图”和“多轮辩论阶段”，避免一次迁移破坏现有配置。
- DAG 不支持回边。角色之间的反复质询由 phase 轮次和跨轮历史实现，避免执行死循环、无限成本和不可预测终止条件。
- 前端采用 `@xyflow/react` 处理节点拖拽、连线、缩放和可访问交互，领域校验与执行仍由 FinBot 后端负责。

## 兼容与回滚

- 旧配置无需迁移脚本；缺失 workflow 时在读取阶段生成。
- 回滚前端不会损坏 `roles/phases/chair`；旧版本会忽略新增 workflow 字段。
- 如需运行态回滚，可恢复 P2 前的 `ai_sites.json` 备份；新字段不涉及数据库 schema。
